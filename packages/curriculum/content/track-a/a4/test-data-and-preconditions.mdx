## Your test is only as good as the data you feed it

A perfectly written test step proves nothing if the data is wrong. Feed a "card declined" test a card that actually succeeds and you've tested the wrong thing while reporting a pass. Test data is not an afterthought you scrape together at run time. It is part of the design.

## See it — a checkout data sheet

Stop inventing data at the keyboard. Design it up front in a sheet, labelled by what each row is meant to prove.

| Label | Card number | Expiry | Coupon | Expected result |
|---|---|---|---|---|
| Valid card, valid coupon | 4242...4242 | 12/30 | SAVE10 | Order placed, 10% off |
| Declined card | 4000...0002 | 12/30 | (none) | "Card declined" error |
| Expired card | 4242...4242 | 01/20 | (none) | "Card expired" error |
| Expired coupon | 4242...4242 | 12/30 | OLD2021 | Coupon rejected, full price |
| Boundary: coupon min spend | 4242...4242 | 12/30 | MIN50 | Accepted at exactly $50.00 |

Every row has a reason to exist. Valid proves the happy path; declined and expired prove error handling; the boundary row pins the exact threshold where behaviour flips.

## Try it — spot the data-dependency trap

A teammate's test "Apply coupon SAVE10" passes on Monday and fails on Tuesday with "coupon already used". Nothing in the code changed.

Reason it through. The coupon is single-use per account, and the test reuses one shared account. Monday's run consumed it; Tuesday's run inherited the used-up state. The failure is real, but it's a data problem, not a bug. The fix is not to "re-enable the coupon by hand" before each run. That's fragile. The fix is to make the test create a fresh account (or fresh coupon) it owns, so it never depends on what came before.

## Do it — harden a BuggyShop test

Open BuggyShop and pick the "checkout with discount" flow. Write the case, but this time write the precondition line explicitly first: what user, what cart contents, what coupon state must be true before step 1. Then build a small data sheet with at least three rows: one valid, one invalid (declined card), one boundary (cart total exactly at a free-shipping threshold). For each row, state the expected result before you run it.

Now stress the self-containment: run your valid row twice in a row. If the second run behaves differently, you've found a data-dependency trap. Note what state leaked between runs and how you'd make the test own its own data instead.
