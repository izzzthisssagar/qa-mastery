## How much can you see inside?

Approaches differ by **how much of the internals you know**:

- **Black box** — you see only inputs and outputs. You test against requirements,
  not code. Most manual testing is here.
- **White box** — you see the code and test its internal paths, branches, and
  conditions. Mostly developers (unit tests, coverage).
- **Grey box** — partial knowledge: you know the API contract, the database
  schema, or the logs, and use that to design sharper tests from the outside.

## See it — the same login, three transparencies

Imagine sliding a dial from black to white on a login feature:

| Setting | What you can see | Example test |
| --- | --- | --- |
| Black | Just the screen | Enter wrong password → expect error |
| Grey | + the API + DB | Confirm a failed login doesn't create a session row |
| White | + the source code | Exercise the branch where the account is locked after N tries |

*(A future "transparency box" widget will let you slide between settings and see
what becomes testable.)*

## Try it — why grey box is a superpower for manual testers

Pure black box can miss things you can't see from the UI. The moment you can read
the network calls or peek at the database, you design tests the UI alone would
never suggest — like confirming a "deleted" record is actually gone, not just
hidden. You don't need to be a developer; a little internal knowledge goes far.

## Do it — classify eight scenarios

For each, decide black / white / grey:
1. Testing checkout purely through the UI.
2. Reading the code coverage report to find untested branches.
3. Checking the API returns 401 for a bad token while testing the login screen.
4. Inspecting the DB to confirm an order row was created.

## Prove it

Five questions on the three approaches.
