## Test fewer values, find the same bugs

You can't test every possible input — a name field alone has billions of values.
So group them. **Equivalence partitioning** splits the inputs into *classes the
system should treat the same way*, then tests **one representative per class**.

If the code accepts `42` in the valid range, it almost certainly accepts `43` the
same way. Testing both adds cost, not coverage. The skill is choosing the classes.

## See it — partitions of one field

Take an age field that accepts **18–60**. Three classes fall out:

| Partition | Example representative | Expected |
| --- | --- | --- |
| Invalid — too low (< 18) | **10** | rejected |
| Valid (18–60) | **35** | accepted |
| Invalid — too high (> 60) | **80** | rejected |

Three tests cover the whole field. Notice these are *behavioural* classes: every
value in "too low" should be rejected for the same reason, so one stands in for
all of them.

> Partitions and boundaries are partners. EP finds the classes; **Boundary Value
> Analysis (A3.3)** then tests the edges *between* them. Do EP first — there are no
> boundaries until you know where one partition ends and the next begins.

## Try it — partition a non-numeric field

Numbers are easy. The real power is that EP works on **anything with classes**.

An **email** field has two behavioural classes: addresses that are **valid** and
addresses that are **invalid**. The invalid class has many shapes — no `@`, two
`@`, no domain, consecutive dots — but they should *all* be rejected. So your
representatives might be:

- Valid: `sagar@example.com`
- Invalid: `not-an-email` (no `@`), `user@@domain..com` (double `@`, double dot)

A correct validator rejects **every** member of the invalid class. If even one
slips through, the partition isn't really being enforced.
