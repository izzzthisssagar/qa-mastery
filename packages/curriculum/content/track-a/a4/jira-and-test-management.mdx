## Why every tester lives in two tools

You will not email spreadsheets of bugs to developers. On a real team, work lives in a tracker (almost always Jira) and test results live in a test management tool (Zephyr, Xray, TestRail, qase). Knowing how these fit together is the difference between "I found a bug" and "here is the case that failed, the build it failed on, and the linked ticket the dev can pick up." This lesson is conceptual — no accounts needed.

## See it — Jira in three layers

Jira tracks units of work called *issues*. The common types:

| Issue type | What it is | Example |
|---|---|---|
| Epic | A large goal spanning weeks | "Checkout redesign" |
| Story | One user-facing slice of an epic | "User can apply a discount code" |
| Bug | A defect in existing behaviour | "Discount code field accepts expired codes" |

Each issue moves through *workflow states* on a *board*: To Do, In Progress, In Review, Done. A board is just a visual column-per-state view of the team's issues. As a tester you mostly file bugs, watch stories move to "In Review" (your cue to test), and drag your verified bugs to Done.

## Try it — read a test run

A *test case* is a reusable set of steps with an expected result. A *test run* (or test cycle) is one execution of a chosen set of cases against a specific build. You mark each case Pass, Fail, or Blocked.

Run "Sprint 12 Regression" against build 3.4.0:

| Case | Title | Result |
|---|---|---|
| TC-07 | Apply valid discount code | Pass |
| TC-08 | Reject expired discount code | Fail |

TC-08 failed: an expired code still applied a discount. That single Fail is your signal to raise a bug — and to *link it to TC-08* so the trail is intact.

## Do it — walk a 10-case run

On BuggyShop, imagine a run of 10 cases TC-01 through TC-10 against the current build.

1. Execute each case and mark its result. Say eight Pass, one Blocked (TC-05 needs data you do not have), one Fail (TC-08, expired code applies).
2. For the Fail, create a Bug in Jira: clear title, steps to reproduce, expected vs actual, the build number.
3. Link the bug to the failed run/case (most tools have a "Create defect" button right on the failed case that does this for you).
4. Update your RTM: the requirement "Reject expired codes" now maps to TC-08, which maps to the new bug — coverage is proven and the defect is traceable.
5. Leave TC-05 as Blocked with a note; a blocked case is not a pass and must not be silently skipped.
