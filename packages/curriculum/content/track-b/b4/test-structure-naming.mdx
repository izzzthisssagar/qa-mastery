## A test should read like a sentence

Anyone — including you in six months — should glance at a test and know what it
proves. Structure and naming are what make that possible. They cost nothing to
do and pay off every time a test goes red.

## See it — Arrange, Act, Assert

Every test has three beats. Keep them visually separate:

```java
@Test
void loginWithValidCredentials_redirectsToDashboard() {
    // Arrange
    loginPage.open();

    // Act
    loginPage.loginAs("standard_user", "secret_sauce");

    // Assert
    assertEquals("Dashboard", dashboardPage.getHeading());
}
```

Arrange sets up. Act does the **one** thing under test. Assert checks the
result. Read top to bottom, it is a sentence.

## Try it — one behaviour per test

Resist cramming. A test named `testLogin` that checks the redirect, the welcome
banner, the cart count, and the logout link is four tests pretending to be one.
When it fails on assertion two, you never learn about three and four. Split it.
Each test should fail for exactly one reason — that is how a red result becomes a
diagnosis instead of a mystery.

## Do it — a written exercise

Take a vague test named `test1` that does several things. Rewrite it as a set of
focused tests. For each, (1) give it a name in the
`scenario_expectedResult` style, (2) mark the Arrange / Act / Assert blocks, and
(3) confirm it has a single clear assertion focus. Then check independence: does
any of your tests rely on data or login state left by another? Describe how you
would make each one stand alone so the suite passes in any order.
