## Testing is not "clicking around"

Ask ten people what a tester does and most will say "they try to break the app."
Closer, but still off. The real job:

> **Testing is gathering information about quality and risk so someone can make
> an informed decision about whether to ship.**

You are not there to bless a release or to prove it's perfect. You're there to
shine a light on what works, what doesn't, and what's still unknown — so the
decision to ship (or not) is made with eyes open instead of crossed fingers.

## See it — the ship / don't-ship call

Imagine a release dashboard the night before launch:

| Signal | State |
| --- | --- |
| Checkout — happy path | ✅ tested, passing |
| Checkout — expired card | ❓ never tested |
| Login | ✅ tested, passing |
| Refunds | ❌ tested, failing |

Should you ship? There's no single right answer — but notice the tester's value
isn't the ✅s, it's surfacing the ❓ and the ❌. The unknown (expired card) is
risk nobody priced in. *(A future interactive "ship-or-don't-ship" simulator will
let you make this call and watch the consequences play out.)*

## Try it — three words people confuse

- **Quality Assurance (QA)** — improving the *process* so defects are less likely
  in the first place (standards, reviews, definition-of-done).
- **Quality Control (QC)** — checking the *product* for defects. Testing lives here.
- **Testing** — actually exercising or examining the product to find where it
  behaves unexpectedly.

So "QA tester" is a slight contradiction the industry made peace with: most
"QA testers" spend their day doing QC.

## Do it — reframe a bug you've seen

Think of the last piece of software that annoyed you. Write one sentence in the
form: *"When I did X, I expected Y, but it did Z."* That single sentence — the
gap between expected and actual — is the atom of everything in this track.
