## Stop repeating yourself

By now your tests have steps that recur — log in, add to cart, read a price. A
**method** lets you name that step once and call it everywhere. That's the whole
point: less copy-paste, fewer places for a bug to hide.

## See it — a method that returns a boolean

```java
public boolean loginSucceeded(WebDriver driver) {
    return driver.findElements(By.id("welcome")).size() > 0;
}
```

Read it right to left of the name: `boolean` is the **return type** (what it hands
back), `loginSucceeded` is the name, and `(WebDriver driver)` is a **parameter** —
an input the caller supplies. The body computes a `boolean` and `return`s it. Now
a test reads cleanly:

```java
Assert.assertTrue(loginSucceeded(driver), "Expected to be logged in");
```

The check lives in one method; fix it once, every test benefits.

## Try it — class vs object

A **class** is a blueprint. An **object** is a thing built from it. Picture a Page
Object — the pattern you'll use constantly in Selenium:

```java
public class LoginPage {
    String url;                      // a field (state)

    public LoginPage(String url) {   // constructor
        this.url = url;
    }

    public void open() {             // a method (behaviour)
        System.out.println("Opening " + url);
    }
}
```

`url` is a **field** — data each object carries. The **constructor**
(`LoginPage(String url)`) runs when you build one. You create objects with `new`:

```java
LoginPage login = new LoginPage("http://localhost:3001/login");
login.open();   // Opening http://localhost:3001/login
```

One class, as many objects as you need — each with its own field values.
