## Results nobody can read are results nobody trusts

A passing build is a claim. A report is the evidence. When a test fails at 2am in
CI, the person who looks the next morning needs to see *what* failed, *why*, and
*what the page looked like* — without rerunning anything. Reporting and logging
are how the suite explains itself.

## See it — reports at a glance

Two common reporters in the Java world:

```
  ExtentReports  ── HTML dashboard: pass/fail, timings, steps, screenshots
  Allure         ── HTML report grouped by feature + severity, history trends
```

Both attach screenshots and logs to each test. You wire them into your TestNG
listeners once, and every run produces a shareable report. The detail is what
turns "12 failed" into "these 12 failed, here's each one's last screen."

## Try it — screenshot on failure

Hook a TestNG `ITestListener` so a screenshot is grabbed the instant a test
fails:

```java
public void onTestFailure(ITestResult result) {
    File shot = ((TakesScreenshot) driver).getScreenshotAs(OutputType.FILE);
    // copy into the report's output folder, attach to the failed test
}
```

Now the report shows the actual page at the moment of failure — usually the
fastest route to the cause.

## Do it — a written exercise

Sketch the logging and reporting plan for a small suite. (1) List the four Log4j2
levels and write one example message you would log at each during a login test.
(2) Decide what your report must show for a failed test to be diagnosable
without a rerun — name at least three things. (3) Explain in two sentences why a
screenshot captured *on failure* beats reading the stack trace alone. Keep it
practical: imagine handing the report to a teammate who did not run the suite.
