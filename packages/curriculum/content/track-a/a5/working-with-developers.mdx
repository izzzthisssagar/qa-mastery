## The bug tracker is not a battlefield

You will find real bugs that developers swear are "not a bug." This is normal. The tester who wins these exchanges is not the loudest one — it is the one with the cleanest evidence and the calmest tone. Your job is to make the truth easy to see, not to be right at someone.

## See it — same fact, two framings

A login bug, written two ways:

| Hostile version | Professional, evidence-led version |
| --- | --- |
| "Login is broken AGAIN. Did anyone even test this?" | "Login fails on build 4.2.1, Chrome 125, staging. Steps below. Expected: dashboard. Actual: 500 error. Video + network log attached." |

Same defect. The left one starts a war and gets ignored. The right one is hard to dispute and gets fixed.

The difference is not politeness theatre. The right version carries **build, environment, steps, expected vs actual, and proof** — the dev can act on it without asking you a single follow-up question.

## Try it — defuse a "Not a bug"

A developer closes your ticket: *"Not a bug. Cart total is correct on my machine."*

Walk through the response:

1. **Assume good faith.** They probably tested a different state than you did.
2. **Find the gap.** You used a discount code; their test had none.
3. **Reply with data, not heat:** "Reproduces with code SAVE10 applied, build 4.2.1. Subtotal $50, expected $45, actual $50 — discount not subtracted. 30s video attached." 
4. **Make it re-openable.** Now the dev can reproduce in one read. No "works on mine" standoff, because you supplied the exact conditions theirs was missing.

You did not argue. You closed the context gap.

## Do it — rewrite two hostile comments

Open BuggyShop, find any two real defects, and for each, write a deliberately hostile comment first — then rewrite it professionally.

Rewrite both of these hostile comments into evidence-led ones (invent realistic build/env details for BuggyShop):

- "The checkout button STILL doesn't work. This is the third time. Useless."
- "Whoever wrote the search feature clearly never tested it. It returns garbage."

For each rewrite, include: build/version, environment, steps to reproduce, expected vs actual, and what proof you would attach. Notice how the personal attack disappears once the facts are present — there is no room left for it.
