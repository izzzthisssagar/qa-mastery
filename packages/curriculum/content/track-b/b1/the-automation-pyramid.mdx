## A shape that tells you where to put your effort

Not all automated tests cost the same. The **automation pyramid** is a rule of
thumb for the *proportions*: write many cheap, fast tests low down and few
expensive, slow ones up top. Get the shape right and your suite runs in minutes
and tells you exactly what broke. Get it upside-down and it runs in an hour,
fails at random, and shrugs when you ask where the bug is.

## See it — the three layers

```
        /\        E2E (UI)   — few, slow, whole-stack
       /  \
      /----\      Integration — some, medium speed
     /      \
    /--------\    Unit        — many, fast, one function
   /__________\
```

- **Unit** (wide base): test one function in isolation. Thousands run in
  seconds; a failure points at a single place.
- **Integration** (middle): check that pieces work *together* — a function and
  its database, two modules at their seam.
- **End-to-end** (narrow top): drive the real app through a real browser, the
  way Selenium does. Closest to the user, slowest to run, most likely to flake.

## Try it — spot the cost trade-off

A unit test might run in **2 milliseconds**; an e2e test in **20 seconds** —
ten thousand times slower, and it touches the network, the browser, and the DB,
so any of those can make it fail for reasons that have nothing to do with your
code. That is why e2e tests are kept few and reserved for **critical journeys**
(can a user check out?), never for logic a unit test could verify instantly.

## Do it — a written exercise

The **ice-cream cone** is the pyramid flipped: a mountain of brittle UI tests
balanced on almost no unit tests. Write down two concrete problems a team would
feel living with one — think about run time, flakiness, and how clearly a
failure tells you where the bug is. Then take one of your manual test cases and
decide which layer it belongs in: would a unit test at the base catch this
faster and more reliably than an e2e test at the top? Justify your choice in a
sentence, mentioning **maintenance** cost.
