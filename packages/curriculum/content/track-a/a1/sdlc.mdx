## The cycle every product rides

**SDLC** — the Software Development Life Cycle — is the set of phases software
moves through from "someone has an idea" to "it's running in production." Knowing
it tells you *when* a tester adds value, which is most of the job.

The classic phases:

1. **Requirements** — what are we building, and why?
2. **Design** — how will it be built (architecture, UI)?
3. **Development** — write the code.
4. **Testing** — does it meet the requirements?
5. **Deployment** — ship it to users.
6. **Maintenance** — fix, improve, monitor.

## See it — QA isn't a phase, it's a thread

A junior tester thinks "QA = phase 4." A good tester works in every phase:

| Phase | What QA does |
| --- | --- |
| Requirements | Review for ambiguity, testability, missing cases |
| Design | Ask "how would I test this?", spot risky areas |
| Development | Write test cases, pair on edge cases |
| Testing | Execute, report, retest |
| Deployment | Smoke test, sign-off |
| Maintenance | Regression, monitor production |

The earlier QA engages, the cheaper the defect — a theme you'll meet again in A1.5.

## Try it — three ways to run the cycle

- **Waterfall** — each phase done once, fully, in order. Testing is a big phase
  near the end. Simple, but defects found late are expensive.
- **Agile** — the cycle repeats in short sprints (1–2 weeks); a slice of the
  product is requirements→tested→shippable each time. Testing happens *every*
  iteration.
- **DevOps** — Agile plus heavy automation of build, test, and deploy, so changes
  flow to production continuously.

phases and toggle Waterfall / Agile / DevOps to watch the flow reshape.)*
