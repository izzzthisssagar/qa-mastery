## A branch is a safe place to work

If everyone edited the same line of history at once, the project would be chaos.
A **branch** is Git's answer: a separate line of work that starts from a point in
history and moves on its own. The default branch is usually called **main**. You
create a branch, do your work there, and `main` stays untouched until you decide
to bring the work back.

## See it — why feature branches

The common pattern is one branch per piece of work — a **feature branch**:

```
  main      A───B───C
                 \
  add-cart-test   D───E   (your branch)
```

You branched at `B`, added commits `D` and `E` (your new cart test) while `main`
stayed at `C`. Your half-done test never destabilises the shared branch. If a
release ships from `main` today, your unfinished work is not in it. This is why
testers keep each new test or fix on its own branch.

## Try it — merging back

When the work is ready, you **merge** it into `main`. Git takes the commits from
your branch and combines them in:

```
  git switch main            # go to the branch you merge INTO
  git merge add-cart-test    # bring your branch's commits in
```

Now `main` has your cart test plus everything else. If nobody touched the same
lines, Git merges cleanly and you are done.

## Resolving conflicts calmly

A **merge conflict** is not an error you broke — it is Git asking a question.
It happens when two branches changed the *same lines*. Git marks the file:

```
  <<<<<<< HEAD
  driver.findElement(By.id("submit"));
  =======
  driver.findElement(By.cssSelector("#submit"));
  >>>>>>> add-cart-test
```

You decide which version is correct (or write a blend), delete the marker lines,
save, then `git add` the file and commit. That is the whole ritual. Conflicts
feel scary the first time; they are routine once you know they are just "pick the
right lines."
