## Put it all together

Everything in this track converges here: a small but real framework that logs in
and checks out on a practice site. The goal is not size — it is a clean
structure you could show in an interview and say "I built this, here is why each
part exists."

## See it — the shape of the framework

```
  src/test/java/
    pages/      LoginPage, InventoryPage, CheckoutPage   (actions, not raw locators)
    tests/      LoginTest, CheckoutTest                   (Arrange-Act-Assert)
    base/       BaseTest (driver setup/teardown, headless flag)
  pom.xml       Selenium, TestNG, WebDriverManager, reporter
  .github/workflows/tests.yml   runs mvn test headless on push
```

Page Objects hold the locators; tests read like sentences; Maven builds it;
GitHub Actions runs it. Each layer has exactly one job.

## Try it — the SauceDemo flow

Target [saucedemo.com](https://www.saucedemo.com). The end-to-end path:

```
  login(standard_user) -> add backpack to cart -> checkout
  -> fill name + zip -> finish -> assert "Thank you for your order!"
```

`CheckoutTest` drives `LoginPage`, `InventoryPage`, then `CheckoutPage`, and
ends on one clear assertion: the confirmation text.

## Do it — a written exercise

Write the build plan for this capstone. (1) List your Page Object classes and,
for each, the actions it exposes (not the locators). (2) Outline the two tests —
a login test and the full checkout test — marking Arrange/Act/Assert and the one
assertion each ends on. (3) State your stable locator strategy and where waits
go. (4) Write your **definition of done**: passes headless in CI, independent
named tests, a report with screenshot-on-failure, and a README. That checklist
is what separates a demo from a framework.
