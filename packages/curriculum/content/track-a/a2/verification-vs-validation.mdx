## Two questions that sound the same

Beginners mix these up forever, so anchor them now:

- **Verification — "are we building the thing right?"** Does the product match
  its specification, design, and standards?
- **Validation — "are we building the right thing?"** Does the product actually
  meet the user's real need, spec or no spec?

You can pass verification and fail validation: a feature built *exactly* to a spec
that was wrong in the first place. That's why both exist.

## See it — same feature, two lenses

Take "Login":

| Lens | Question | Activity |
| --- | --- | --- |
| Verification | Does it match the spec? | Review the login spec, check the build against each requirement |
| Validation | Does it serve the user? | Actually log in; is the flow usable, does it solve the real need? |

*(A future split-screen widget will show one side reviewing the spec while the
other tests the running build.)*

## Try it — static vs dynamic

- **Static testing** examines artifacts *without running* the software — reviews,
  walkthroughs, inspections of requirements, designs, and code. This is mostly
  *verification*, and it's the cheapest place to catch defects.
- **Dynamic testing** runs the software with inputs and checks the behaviour.
  This is mostly *validation*.

A document review that finds a contradictory requirement just prevented a defect
before a line of code existed — that's static verification earning its keep.

## Do it — sort the activities

Label each as verification (V) or validation (Val):
1. Inspecting the requirements document for gaps.
2. Logging into the built app and checking the flow feels right.
3. A code review against the design.
4. Running the checkout end-to-end as a user would.
