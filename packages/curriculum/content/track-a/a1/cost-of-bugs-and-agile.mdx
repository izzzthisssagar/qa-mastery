## A bug gets more expensive every day it survives

The single most useful number in QA folklore: a defect caught in **requirements**
might cost \$1 to fix; the same defect caught in **production** can cost 100×.
Each phase it slips through, more work is built on top of it.

| Caught in… | Relative cost |
| --- | --- |
| Requirements | 1× |
| Development | ~5× |
| Testing | ~10× |
| Production | ~100× |

*(A future "catch the bug" game will let you catch the same defect at each phase
and watch the cost multiply.)*

## See it — why "shift left" wins

If late defects are the expensive ones, the move is obvious: **catch them
earlier.** "Shift left" means pushing testing activities toward the start —
reviewing requirements, questioning designs, writing cases before the build is
done. The cheapest test is the one that finds a defect before code exists.

## Try it — QA's seat in Scrum

Agile doesn't have a "testing phase" to hide in. QA shows up across the ceremonies:

- **Backlog refinement** — "this story has no acceptance criteria — how do we test it?"
- **Sprint planning** — size the testing, not just the coding.
- **Daily standup** — flag blockers (no test data, broken environment).
- **The sprint** — test stories as they become ready, not all at the end.
- **Review & retro** — demo tested work; improve how testing happened.

## Do it — five sprint judgment calls

For each, decide what a good QA does:
1. A story is marked "done" but has no acceptance criteria.
2. A dev says "it's a tiny change, no need to test."
3. The test environment is down on day 2 of the sprint.
4. A critical bug is found the afternoon before the demo.
5. The same regression keeps reappearing every release.
