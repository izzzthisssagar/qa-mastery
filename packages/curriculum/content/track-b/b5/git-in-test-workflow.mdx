## Git is where testing and code meet

The earlier lessons were Git mechanics. This one is the tester's payoff: Git is
the seam where your tests and the application code live together, get reviewed
together, and get verified automatically. A good Git habit makes a test suite
trustworthy; a sloppy one fills history with noise and lets broken code slip in.

## See it — .gitignore for test artifacts

Running tests produces junk you never want in history: compiled output,
screenshots from failures, HTML reports, log files. You list these in a
**.gitignore** file and Git stops tracking them:

```
  # .gitignore
  target/
  test-output/
  *.log
  screenshots/
  allure-results/
```

Now `git status` shows only real changes — your test code and config — not a
hundred generated files. Clean diffs make review fast, which is the whole point.

## Try it — commit tests with the code

When you fix a bug or add a feature, commit the **test alongside it**. The fix
and the test that proves it belong in the same branch and ideally the same pull
request:

```
  git add CheckoutService.java CheckoutServiceTest.java
  git commit -m "fix tax rounding + regression test"
```

A reviewer then sees both at once: the change *and* the evidence it works. The
test can never quietly fall out of sync with the code it covers, because they
moved through history together.

## Branches per feature, CI on push

The workflow ties together everything from this module. Each feature or fix gets
its **own branch** carrying its code and tests. You push the branch and open a
pull request. On that push, **CI** (continuous integration — a server like GitHub
Actions) automatically runs the whole suite. If a test fails, the PR shows red
and the merge is blocked:

```
  push branch  ->  CI runs tests  ->  green = mergeable / red = blocked
```

This is the safety net. Nobody has to remember to run the suite; the server does
it on every push, so a regression is caught at the PR, not by the next teammate
who pulls a broken `main`.
