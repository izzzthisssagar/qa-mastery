## Four checks people constantly confuse

"Smoke," "sanity," "retest," "regression" get thrown around as if they mean the same thing. They don't. Each answers a different question at a different moment in a release. Mix them up and you either waste hours or ship a build that's broken in ways you already knew how to catch.

## See it — what each one asks

| Check | Scope | Question it answers | When it runs |
|---|---|---|---|
| Smoke | Wide, shallow | Is this build stable enough to test at all? | First, on every new build |
| Sanity | Narrow, deep | Does this specific fix/area actually work? | After a targeted change, before full testing |
| Retest | One case | Is *this exact bug* fixed? | When a dev marks a defect resolved |
| Regression | Broad suite | Did this change break anything elsewhere? | Before release, after changes settle |

Smoke and sanity decide whether to bother. Retest and regression decide whether to ship.

## Try it — walk a release through all four

BuggyShop v1.1 adds a coupon field to checkout. Trace the flow:

1. **Smoke** the new build: can you log in, add to cart, reach checkout, and pay? If checkout 500s, you stop — no point testing coupons on a dead build.
2. **Sanity** the coupon area: valid code applies a discount, invalid code is rejected, expired code is rejected. Narrow, but you go deep on the thing that changed.
3. **Retest** BS-014, the old "cart total ignores quantity" bug the devs say they fixed. Re-run the exact reproduction steps. Fixed? Close it.
4. **Regression**: run the broader cart/checkout suite. The coupon change touched the totals calculation — does free shipping still apply? Does tax still round correctly?

That regression run is where you find BS-021 ("coupon stacks with itself") and BS-022 ("total goes negative on a 100%-off code"). Neither was the feature under test — they're collateral damage the broad suite caught.

## Do it — grow the suite

In BuggyShop, pick any two bugs you've already filed and closed. For each, write one concise regression case: preconditions, steps, expected result. Add them to a list called "Regression — Cart." That list is your suite, and it should get longer every release. The rule: once a bug ships and gets fixed, its test lives forever, so the same break can't sneak back.
