## Tests only protect you if they actually run

A suite that runs only when someone remembers to run it protects nothing. **CI**
runs it for you — on every push, on a clean machine, the same way every time.
GitHub Actions is the common choice and it is just a YAML file in your repo.

## See it — a workflow on every push

```yaml
name: tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: '17' }
      - run: mvn -B test
      - uses: actions/upload-artifact@v4
        if: always()
        with: { name: reports, path: target/surefire-reports }
```

On every push and PR, GitHub spins up a fresh Ubuntu runner, installs Java, runs
`mvn test`, and uploads the reports. No screen, no manual step.

## Try it — headless and artifacts

The runner has no display, so the browser must run **headless**:

```java
ChromeOptions options = new ChromeOptions();
options.addArguments("--headless=new");
```

And `if: always()` on the upload step matters — you want the report saved
*especially* when tests fail. That report and any screenshots become
**artifacts** you download from the run to diagnose the break.

## Do it — a written exercise

Plan CI for a Maven + TestNG suite. (1) Name the events that should trigger a
run and why a pull request is the key one. (2) Explain what must change in your
WebDriver setup for the suite to work on a server with no screen. (3) Decide
which artifacts you would upload and why you'd save them even on failure.
Finally, describe **fail-fast**: what should happen the moment a test step fails,
and how that protects the main branch from a broken merge.
