## Variables are typed boxes

In Java, every variable is declared with a **type**, and that type is fixed.
`int count = 5;` says "this box holds whole numbers." Try to put text in it and
the compiler rejects you before the test ever runs — which is exactly the safety
you want when automating.

There are two families. **Primitives** hold a raw value directly: `int`, `double`
(decimals), `boolean` (true/false), `char`. **Objects** are references — the
variable holds a pointer to data living elsewhere. `String`, the type for text,
is an object, and that distinction bites testers constantly.

## See it — declaring the types you'll actually use

```java
String pageTitle = "BuggyShop";   // object: text
int itemCount = 3;                // primitive: whole number
double price = 19.99;             // primitive: decimal
boolean isLoggedIn = false;       // primitive: true/false
```

`isLoggedIn` is the one to remember — Selenium checks like `isDisplayed()` or
`isSelected()` return a `boolean`, and your assertions live or die on it.

## Try it — the String trap

```java
String a = getDriverTitle();      // imagine this returns "BuggyShop"
boolean wrong = (a == "BuggyShop");        // compares references — unreliable
boolean right = a.equals("BuggyShop");     // compares contents — correct
```

`==` asks "are these the *same object*?" For text you want "do they *say* the
same thing?" — that's `.equals()`. Using `==` on Strings is one of the most common
bugs in a beginner's test suite: it sometimes passes by luck and then fails
mysteriously.

## Casting and operators

Mixing number types needs **casting**. Widening is automatic
(`double d = itemCount;`); narrowing must be explicit and truncates:
`int n = (int) 3.9;` gives `3`. Operators are what you'd expect: arithmetic
(`+ - * /`), comparison (`== != < >`), and logical (`&&`, `||`, `!`) — the last
group produces the booleans your assertions feed on.
