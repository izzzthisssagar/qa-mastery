## When conditions gang up on you

Some bugs only appear when several things are true at once. A coupon that should stack with a member discount but doesn't. Free shipping that fires for guests it shouldn't. Testing each condition alone misses these, because the defect lives in the *combination*. A decision table is the technique that systematically covers those combinations so none slip through.

## See it — conditions as rules

BuggyShop offers free shipping when **all three** hold: cart total is over the threshold, the user is a member, AND a valid promo code is applied. Three yes/no conditions give 2 x 2 x 2 = 8 combinations. Lay them out as columns:

| Condition / Rule | R1 | R2 | R3 | R4 | R5 | R6 | R7 | R8 |
|---|---|---|---|---|---|---|---|---|
| Cart over threshold | Y | Y | Y | Y | N | N | N | N |
| Is member | Y | Y | N | N | Y | Y | N | N |
| Valid promo | Y | N | Y | N | Y | N | Y | N |
| **Action: free shipping** | Yes | No | No | No | No | No | No | No |

Only R1 grants free shipping. That single "Yes" surrounded by "No" is exactly the kind of logic a developer gets subtly wrong.

## Try it — collapse with don't-cares

Look at R5 through R8: the cart is below the threshold, so shipping is never free no matter what the other conditions say. The member and promo values are **don't-cares** here. Collapse them into one rule:

| Condition / Rule | R1 | R2 | R3 | R4 | R5* |
|---|---|---|---|---|---|
| Cart over threshold | Y | Y | Y | Y | N |
| Is member | Y | Y | N | N | - |
| Valid promo | Y | N | Y | N | - |
| **Free shipping** | Yes | No | No | No | No |

Eight columns became five. You dropped four redundant tests without losing any meaningful coverage. The `-` means "value doesn't affect the outcome." Always sanity-check a collapse: ask "could flipping this condition ever change the action?" If yes, you can't collapse it.

## Do it — build one for payment availability

On BuggyShop, "Pay on delivery" is offered only when the order is **domestic** AND under a value cap AND the customer's account is **verified**. Write the full table: three conditions, list all eight combinations as columns, fill in the action row (offered / not offered). Then collapse any don't-cares — if the order is international, the other two conditions stop mattering. Finally, write one test case per surviving rule, each with concrete inputs (a real cart, a real account state) and the expected availability. Compare your count to a naive "test each condition once" approach; you'll have caught combinations that approach ignores.
