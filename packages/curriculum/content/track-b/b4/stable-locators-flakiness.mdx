## A test that breaks on good code is your fault, not the app's

Flakiness is the tax you pay for fragile tests. The two biggest sources are
**bad locators** and **bad waiting**. Fix those and most "it failed for no
reason" reports disappear. This lesson is the habit set that keeps a suite
trustworthy.

## See it — resilient locators

Selenium finds elements by a locator. Some locators survive change; some shatter
the moment a designer nudges the layout.

```
  GOOD   By.id("login-btn")               stable, unique
  GOOD   By.cssSelector("[data-testid='login']")  test-only hook
  OK     By.cssSelector("button.primary[type=submit]")  attribute-based
  BAD    By.xpath("/html/body/div[3]/form/button[1]")    position-based
```

Prefer an `id` or a `data-testid` the developers add for you. Avoid absolute
XPath and avoid locators that lean on visible text that copywriters change.

## Try it — kill the sleep

Never paper over timing with `Thread.sleep(3000)`. Wait for the *condition* you
actually care about:

```java
new WebDriverWait(driver, Duration.ofSeconds(10))
    .until(ExpectedConditions.elementToBeClickable(By.id("login-btn")))
    .click();
```

This returns the instant the button is clickable — faster than a fixed sleep on
a good day, and reliable on a slow one.

## Do it — a written exercise

Take one flaky test you have seen or can imagine. Write down (1) the locator it
uses and a more stable replacement, (2) where a fixed sleep should become an
explicit wait and which `ExpectedCondition` you'd poll for, and (3) one piece of
shared state it depends on and how you'd make the test set up and tear down its
own data so it becomes idempotent. Finish with a rule for your team: when is a
retry acceptable, and what must you log every time one fires?
