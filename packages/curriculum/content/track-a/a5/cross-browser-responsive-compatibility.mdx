## Your bug only exists on someone else's screen

A page that looks perfect on your laptop can be broken for a quarter of your users. Different browsers use different rendering engines, and the same CSS resizes, reflows, or collapses differently across phone, tablet, and desktop. If you only test the setup on your own machine, you are testing the one configuration least likely to surprise you.

## See it — why engines disagree

Browsers are not interchangeable. Each ships a rendering engine that decides how HTML and CSS turn into pixels.

| Browser | Engine | Common gotcha |
|---|---|---|
| Chrome, Edge | Blink | Permissive; bugs here often pass everywhere |
| Safari (incl. iOS) | WebKit | Date inputs, flexbox gaps, `100vh` on mobile |
| Firefox | Gecko | Form control styling, scrollbar rendering |

On iOS, every browser is WebKit underneath - even "Chrome for iPhone." So a Safari-only bug hits a huge slice of mobile traffic. That is why "works in Chrome" is not the same as "works."

## Try it — diagnose a mobile-only overlap

A user reports the checkout's "Place Order" button is covered by the order summary - but only on their phone. On desktop everything is fine.

Reason through it. The layout uses absolute positioning that assumes a wide screen. At desktop width the summary sits in a side column; at 375px the column stacks on top of the button and the absolute coordinates no longer hold, so the two elements overlap. The bug is real, it is reproducible, and it is invisible at the width you happened to test. The fix lives in the responsive breakpoint, but you only *find* it by shrinking the viewport.

## Do it — checkout at three viewports

Open BuggyShop's checkout flow. Using your browser's device toolbar (DevTools, then the device icon), set the viewport to three widths and walk the full flow at each:

1. Mobile - 375 x 667
2. Tablet - 768 x 1024
3. Desktop - 1280 x 800

At each width, add an item, reach checkout, and check: is every field reachable, is nothing overlapping or cut off, does the primary button stay tappable, and does text wrap instead of overflowing? Log what changes between widths - those transitions are where responsive bugs hide.

Then sketch a small compatibility matrix. Don't list every browser; pick the top 3-4 combos your analytics would show (e.g. iOS Safari mobile, Chrome desktop, Android Chrome mobile). Test those deliberately, sample the rest. A richer interactive viewport simulator is planned for this lab.

## Prove it

Four questions to check you can pick combos and spot responsive failures, not just resize a window.
