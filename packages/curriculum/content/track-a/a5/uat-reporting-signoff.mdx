A green test run does not mean "ship it." Two questions remain: do the people who will live with this software agree it solves their problem (UAT), and can you state the residual risk clearly enough for someone to make a real decision (the report)? This is where testing stops being mechanical and becomes a judgment call you have to defend.

## See it — UAT vs system testing

Same build, two different questions.

| | System testing | UAT |
|---|---|---|
| Question | Does it meet the spec? | Does it solve the real problem? |
| Owner | QA / dev | Business users / product owner |
| Example finding | "Checkout API returns 200 on valid order" | "I can place an order, but I can't apply my net-30 terms like I do today" |
| Passing it means | Built right | Built the right thing |

Both can pass independently. A feature can be bug-free and still fail UAT because it does not fit how people actually work.

## Try it — read a summary line

A report says: *"Checkout: 48/50 cases passed (96%). 2 open defects: 1 high (guest checkout fails on Safari), 1 low (cosmetic alignment)."*

Reason through it. The 96% looks reassuring, but the number alone is meaningless until you read the open defects. A high-severity defect blocking guest checkout on a whole browser is not a rounding error; it is a chunk of real users who cannot buy. The honest summary is not "96%, looks good" but "96%, with one high-severity item that blocks a real user segment." The pass rate frames; the defects decide.

## Do it — outline a v1.1 summary + ship recommendation

Using BuggyShop v1.1, draft a one-page test summary in prose. Include:

- **Scope** — what you tested (cart, checkout, search) and what you deliberately did not (admin panel, deferred).
- **Results** — pass rate per area, stated as passed/total, not just a percentage.
- **Open defects** — list by severity. For each, one line on user impact.
- **Residual risk** — what could still go wrong in production, plainly.
- **Recommendation** — Go, No-go, or Go-with-conditions. If conditional, name the conditions (e.g., "ship if the high-severity Safari defect is hotfixed; the low cosmetic issue can wait").

Keep fact and judgment separate: the data section reports numbers, the recommendation section is your call. A richer interactive report builder is planned for this lab.
