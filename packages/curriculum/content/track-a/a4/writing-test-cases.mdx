## A bad test case wastes everyone's time

A test case is a contract: anyone on the team should be able to pick it up, follow it, and get the same result you would. When a case is vague or tries to do too much, it fails silently in a different way: people skip steps, results become arguments, and bugs slip through. A clean case is boring to read and impossible to misinterpret. That's the goal.

## See it — the anatomy

Every solid test case carries the same six parts:

| Part | Purpose | Example |
| --- | --- | --- |
| ID | Unique reference for tracking | TC-LOGIN-007 |
| Title | One line saying what's verified | Login fails with wrong password |
| Preconditions | State before you start | A registered account exists; user is logged out |
| Steps | Numbered, single actions | 1. Open /login 2. Enter valid email 3. Enter wrong password 4. Click Sign in |
| Test data | The exact inputs used | email: shop@buggy.test, password: Wrong123 |
| Expected result | What must happen, exactly | Stays on /login; error "Invalid email or password" appears below the form; no session created |

Notice the expected result names the page, the exact message text, and the state. There is nothing to interpret.

## Try it — atomic vs bloated

Read this case: "Test the login page — try logging in, check the dashboard loads, then log out and confirm you're back at login."

That's three checks wearing one ID. If it fails, which part broke? You can't tell from a single pass/fail. Split it into three atomic cases: one for successful login, one for dashboard load after login, one for logout. Now each result is a clean signal. The rule: if your title needs the word "and," you probably have two cases.

## Do it — clean up a BuggyShop case

Open the BuggyShop practice app and find this vague case someone left behind:

"Check that login works properly with bad input."

Rewrite it as one atomic case using all six parts. Pick a single scenario — say, an empty password field. Give it an ID, a title that names the one behavior, a precondition (logged out, on the login page), numbered steps, the exact test data (a valid email, blank password), and an expected result that quotes the actual message BuggyShop shows and the page it stays on. If you find yourself describing two outcomes, stop and split it. (An interactive case-builder for this exercise is planned.)
