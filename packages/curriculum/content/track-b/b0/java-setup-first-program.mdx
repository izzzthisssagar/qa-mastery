## Why Java needs a setup step at all

You picked Java because Selenium's most mature bindings are written in it. But
Java is **compiled**, not scripted — so before you write a single line you need
the toolchain that turns your code into something a machine runs. Three names get
thrown around, and testers confuse them constantly.

The **JVM** (Java Virtual Machine) runs compiled bytecode. The **JRE** is the JVM
plus the standard libraries. The **JDK** is the JRE plus the developer tools —
the compiler `javac`, the launcher `java`, and friends. You write automation, so
you install the **JDK**. Java 17 is the project's baseline.

## See it — the smallest program that runs

```java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Tests starting...");
    }
}
```

`public class Hello` must live in a file named `Hello.java`. The line
`public static void main(String[] args)` is the **entry point** — the JVM looks
for that exact signature to know where to begin. `System.out.println(...)` writes
a line to the console. Every Selenium test you ever run starts from a `main` or a
test method that looks a lot like this.

## Try it — compile, then run

Two steps, two commands:

```
javac Hello.java   // compiles -> produces Hello.class (bytecode)
java Hello         // the JVM runs the bytecode
```

`javac` is the compiler; if you have a typo, it fails here and nothing runs.
`java` launches the JVM on the `.class` file. In practice you'll let an IDE like
**IntelliJ IDEA** do both with one click — but knowing the two stages explains
*why* a "compile error" is different from a "test failure": one happens before
your code ever executes.

## Do it — written exercise

On paper, write the full `Hello.java` from memory, then answer: which command
produced `Hello.class`, and which one printed the text? Name the tool (JDK / JRE
/ JVM) responsible for each. If you can explain why an IDE still needs a JDK
installed underneath it, you understand the layering.
