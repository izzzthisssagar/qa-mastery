## A test is a method that checks something

In TestNG, a test is just a Java method tagged with `@Test`. The framework
finds every `@Test` method, runs each one on its own, and tells you which
passed and which failed. You never call these methods yourself — TestNG drives
them. Compared to stuffing everything into a `main` method, this is the whole
point of a framework: structure, isolation, and a report.

## See it — @Test and an assertion

A test without an assertion always passes, which proves nothing. The assertion
is the verdict:

```java
import org.testng.annotations.Test;
import static org.testng.Assert.assertEquals;

public class LoginTest {
    @Test
    public void titleIsCorrect() {
        String actual = "Swag Labs";
        assertEquals(actual, "Swag Labs"); // (actual, expected)
    }
}
```

`assertEquals(actual, expected)` fails the test if the two differ.
`assertTrue(condition)` fails if the condition is false. Argument order matters
for the failure message: TestNG reports "expected X but found Y".

## Try it — hard vs soft asserts

A **hard assert** stops the test the moment it fails — the lines after it never
run. That is fine when later steps depend on the failed one. But to check
several independent things in one test, use a **SoftAssert**: it records each
failure and reports them all at `assertAll()`.

```java
SoftAssert sa = new SoftAssert();
sa.assertEquals(price, 9.99, "price wrong");
sa.assertTrue(addToCart.isDisplayed(), "no cart button");
sa.assertAll(); // reports every collected failure at once
```

Forget `assertAll()` and a soft assert silently passes — a classic bug.

Write, in plain words, what TestNG does that a `main` method cannot: name three
concrete things (isolation per test, pass/fail reporting, continuing after a

different test where a **soft** assert is the better choice — and say what goes
wrong if you swap them.
