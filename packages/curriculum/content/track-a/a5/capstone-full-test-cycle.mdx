## The deliverable that proves you can test

Everything in Track A built to this. BuggyShop is shipping v2.0 with a new coupon feature, and you own the release. You will not just "find bugs" - you will run a full cycle and hand back a recommendation a release manager can act on. This is what a QA role actually looks like.

## See it — the five deliverables

| # | Deliverable | What it shows | Graded by |
|---|-------------|---------------|-----------|
| 1 | Mini test plan | Scope, risks, approach | Rubric |
| 2 | Test cases | Coverage of the feature using EP, BVA, decision tables, state transition | Auto (coverage) |
| 3 | Execution run | Every case run with pass/fail and evidence | Auto (completeness) |
| 4 | Bug reports | Clear, reproducible defects | Auto (match) + rubric |
| 5 | Summary report | Findings + ship/no-ship call | Rubric |

The auto grader checks whether your bugs match the seeded defects, whether your cases cover the coupon requirements, and whether you actually executed everything. The rubric scores judgement: is the plan focused, is the recommendation defensible.

## Try it — pick the right technique per requirement

Coupon rules for v2.0: code must be 6-10 chars; order must be at least $20 to apply; SAVE10 gives 10% off, FREESHIP removes shipping; a coupon can be applied, removed, or rejected.

- "6-10 chars" - that is a range, so **BVA**: test 5, 6, 10, 11. Pair with **EP** for the valid middle.
- "at least $20" - **BVA** on the threshold: $19.99, $20.00, $20.01.
- "type of coupon changes the discount" - two conditions combine, so a **decision table**.
- "applied / removed / rejected" - the cart moves between states, so **state transition**: can you remove a coupon that was never applied?

Notice each requirement points at a technique. That mapping is the heart of the test plan's approach section.

## Do it — draft the test-plan skeleton

In BuggyShop, open the v2.0 release notes and write a one-page plan with three headed sections:

1. **Scope** - In: applying, removing, and validating coupons at checkout. Out: payment gateway, existing catalogue (unchanged). State your assumptions.
2. **Risks** - List three. Example: "Coupon discount could stack with an existing sale and go negative." Rank each high/medium/low.
3. **Approach** - One line per requirement naming the technique you will use, mirroring the Try it mapping.

Keep it to a page. A test plan nobody reads is worthless. (A guided interactive plan builder is planned for this lab.)

## Do it — run your execution pass

A plan is theory; the run is proof. Execute against the current build below: hunt
the seeded defects, file a clean report for each, and watch your coverage of the
seeded set fill in. This is the execution + bug-report deliverable of your
capstone — pair it with the plan above and a short summary report stating your
ship / no-ship call. (The v2.0 coupon defects arrive with that release; for now
you run against the live seeded set.)
