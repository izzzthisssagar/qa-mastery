## Setup and teardown should not live in the test

Every UI test needs a browser open before it and closed after it. Writing that
inside each `@Test` means copy-pasted boilerplate and a leaked browser the day
you forget the cleanup. TestNG's lifecycle annotations move that work into
hooks that run automatically around your tests.

## See it — the per-method hooks

```java
public class LoginTest {
    WebDriver driver;

    @BeforeMethod
    public void setUp() {
        driver = new ChromeDriver();
        driver.get("https://www.saucedemo.com");
    }

    @Test public void canSeeLoginForm()  { /* ... */ }
    @Test public void canSeeLogo()       { /* ... */ }

    @AfterMethod
    public void tearDown() {
        driver.quit();
    }
}
```

`@BeforeMethod` runs before **every** `@Test`; `@AfterMethod` after every one.
Two tests means setUp/tearDown run twice — each test gets a fresh browser.

Use `@BeforeSuite` for expensive one-time setup (a report, a DB connection),
`@BeforeClass` for per-class work, and `@BeforeMethod` for the driver — fresh
browser per test, no state bleeding from one test into the next.

## Do it — a written exercise

Sketch the run order for a class with `@BeforeSuite`, `@BeforeClass`,
`@BeforeMethod`, two `@Test` methods, and their matching after-hooks — list
every line in the exact order it fires. Then explain in one or two sentences
why putting `new ChromeDriver()` in `@BeforeClass` instead of `@BeforeMethod`
would make the two tests share a browser, and one bug that could cause.
