## Version control is a save button with memory

When you write test code or a test plan, you keep editing it. Without version
control, "save" overwrites the old version forever. **Git** is version control:
every time you commit, it records a snapshot you can return to, compare against,
or undo. As a tester you will read history to see *what changed* before a bug
appeared, so this matters even if you never write production code.

## See it — repo, commit, history

A **repository** (repo) is a folder Git is tracking. You turn a plain folder
into one with `git init`. From then on, Git watches the files and lets you save
**commits** — snapshots, each with a short message like `add login smoke test`.
History is just the chain of those commits, newest on top:

```
  commit 3  "fix flaky wait on checkout"   <- newest (HEAD)
  commit 2  "add login smoke test"
  commit 1  "initial test project"
```

## Try it — the staging area

Git does not commit everything you changed automatically. There is a middle
step. You **stage** the exact changes you want with `git add`, then turn the
staged set into a snapshot with `git commit`:

```
  git init                 # start tracking this folder
  git add LoginTest.java   # stage one file
  git commit -m "add login smoke test"
```

The staging area is why one commit can hold *just* the login test while you leave
half-finished work uncommitted. Stage deliberately; a tidy history is easier to
read when you are hunting a regression.

## Working tree vs history

Your **working tree** is the files on disk right now — what your editor shows.
**History** is the committed snapshots. Editing a file changes only the working
tree; nothing is permanent until you commit. `git status` shows what changed but
is not yet committed, so it is the command you run constantly to see where you
stand.
