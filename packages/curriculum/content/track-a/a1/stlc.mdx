## The tester has a lifecycle too

The SDLC is how the *product* gets built. The **STLC** — Software Testing Life
Cycle — is how *testing* gets done, phase by phase, inside it:

1. **Requirement analysis** — what's testable here? what's ambiguous?
2. **Test planning** — scope, approach, who, when, what tools (the test plan).
3. **Test case design** — write the cases and prep test data; build the RTM.
4. **Environment setup** — get a place to run the tests (data, accounts, builds).
5. **Test execution** — run cases, log results, raise defects, retest.
6. **Test closure** — report, capture lessons, archive artifacts.

## See it — entry and exit criteria

Each phase has a gate. You don't start execution with no test cases; you don't
*close* with half the cases unrun. Example for execution:

| | Criteria |
| --- | --- |
| **Entry** | Test cases reviewed; environment ready; build deployed |
| **Exit** | All planned cases run; critical defects fixed & retested; results reported |

Criteria are what stop "are we done?" from being a vibe. *(A future interactive
STLC wheel will let you click each phase to see its real artifacts.)*

## Try it — the artifacts

Match the phase to what it produces:

- Test planning → **Test Plan**
- Test case design → **Test cases + RTM + test data**
- Test execution → **Test logs + defect reports**
- Test closure → **Test summary report**

The **RTM** (Requirements Traceability Matrix) is the quiet hero: it links every
requirement to the cases that cover it, so "did we test everything?" has an answer.

## Do it — order a mini cycle

A spec lands: "BuggyShop wishlist — users can save items and move them to cart."
Put these in STLC order and name one exit criterion for execution: *design cases,
set up a test account, analyse the spec for gaps, run the cases, write the summary.*
