## Your repo and GitHub are two copies

So far Git has lived on your machine. To collaborate, you also keep a copy on a
server like **GitHub**. That remote copy has a name: **origin** by default. Your
local repo and origin are independent — commits you make locally are not on
GitHub until you send them, and commits a teammate pushed are not on your machine
until you fetch them. Keeping the two in sync is the daily rhythm of team work.

## See it — push and pull

Two commands move commits between local and origin:

```
  git push        # send your local commits up to origin
  git pull        # bring origin's commits down and merge them in
```

**Push** shares your finished work; **pull** catches you up with what others
merged. A common habit: pull before you start so you build on the latest code,
push when your branch is ready so it is backed up and visible to the team.

## Try it — fetch vs pull

`git pull` is actually two steps glued together: download, then merge. Sometimes
you want only the first half. `git fetch` downloads the remote commits but does
**not** touch your working branch, so you can look before you leap:

```
  git fetch       # download remote commits, do not merge yet
  git log origin/main   # inspect what changed
  git merge origin/main # merge when you are ready
```

Use `fetch` when you want to see what landed before integrating it; use `pull`
when you just want to be up to date now.

## Pull requests — review before merge

A **pull request** (PR) is a GitHub feature, not a Git command. You push your
feature branch, then open a PR asking to merge it into `main`. The PR shows a
**diff** of every change and a place for reviewers to comment line by line.
Nothing merges until someone approves and the automated checks pass.

For testers this is gold: the PR is where you review whether a change has test
coverage, spot risky edits, and block a merge until the tests are green. It turns
"trust me, it works" into a reviewable, gated record.
