## One test, many inputs

You rarely want to test login with just one username. You want valid users,
locked-out users, wrong passwords, empty fields. Copy-pasting the test method
four times is the wrong answer — four methods to maintain when the form
changes. Data-driven testing writes the logic **once** and feeds it many rows.

## See it — a @DataProvider

```java
@DataProvider(name = "logins")
public Object[][] logins() {
    return new Object[][] {
        { "standard_user", "secret_sauce", true  },
        { "locked_out_user", "secret_sauce", false },
        { "standard_user", "wrong",         false },
    };
}

@Test(dataProvider = "logins")
public void loginOutcome(String user, String pass, boolean shouldPass) {
    loginPage.login(user, pass);
    assertEquals(loginPage.isLoggedIn(), shouldPass);
}
```

The provider returns rows; the test runs **once per row**, with each row's
values bound to its parameters. Three rows, three results in the report.

## Try it — data and logic, kept apart

Notice what changed and what didn't. The login steps — the **logic** — appear
exactly once. The cases — the **data** — live in their own method. To add a test
for an empty password, you add one row, not one method:

```java
{ "standard_user", "", false },
```

That is the whole benefit: the part that changes often (the inputs) is separate
from the part that rarely changes (the steps). Later you can pull the rows from
a CSV or Excel file and the test method never moves.

## Do it — a written exercise

Design a `@DataProvider` for a search box, in plain text: list four input rows
covering a normal word, an empty search, a term with no results, and a term
with special characters — and for each row, what the expected outcome is. Then
explain in one or two sentences why this is better than writing four separate
`@Test` methods, and what you would change to add a fifth case.
