## Tests are too fast for their own good

Most "random" Selenium failures are not random at all — the test asked for an
element a few milliseconds before the page produced it. Modern pages load
content asynchronously, so the cure is not to slow the whole test down but to
make each step **wait for the condition it actually depends on**.

## See it — where flakiness comes from

```java
driver.get("https://the-internet.herokuapp.com/dynamic_loading/1");
driver.findElement(By.cssSelector("#start button")).click();
driver.findElement(By.id("finish")).getText();  // FLAKY: #finish appears later
```

The button kicks off a spinner; `#finish` does not exist until the spinner ends.
The test reaches the last line first and throws `NoSuchElementException`
sometimes — pass, fail, pass — the signature of a synchronization bug.

## Try it — implicit vs explicit waits

An **implicit wait** is one global setting: retry every lookup for up to N
seconds.

```java
driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
```

An **explicit wait** targets one step and one condition:

```java
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
WebElement finish = wait.until(
    ExpectedConditions.visibilityOfElementLocated(By.id("finish")));
System.out.println(finish.getText());  // "Hello World!"
```

`ExpectedConditions` gives you precise triggers: `visibilityOfElementLocated`,
`elementToBeClickable`, `textToBePresentInElement`. The wait returns the moment
the condition is met, so a fast page is not penalised and a slow one is
tolerated.

Never reach for `Thread.sleep(5000)`. It blocks for the full five seconds every
time even when the element is ready instantly, and it *still* fails if the page
takes six. It waits for the clock; explicit waits wait for the page.

## Do it — a written exercise

Take the flaky snippet above and rewrite it in words using a `WebDriverWait`
with `ExpectedConditions.elementToBeClickable` before the click and
`visibilityOfElementLocated` before reading `#finish`. Then explain in two
sentences why replacing the wait with `Thread.sleep(3000)` is the wrong fix —
cover both the slow-when-fast and still-fails-when-slow problems.
