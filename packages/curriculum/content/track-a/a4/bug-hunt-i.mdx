## The exam that proves you can actually test

You have practised graded bug-report labs already — the EP and BVA labs scored your reports field by field. This milestone is different: nobody tells you where the bugs are. You get BuggyShop v1.0, a ~90 minute clock, and the whole signup -> checkout flow. There are 18 seeded bugs. Find and report about 10 to pass.

This is the closest thing to a real testing day you have seen. It rewards the same instincts a hiring manager looks for: coverage under time pressure, clean repro steps, and honest severity.

## See it — how the grade is built

Your score is not "did you write a lot." Each report is checked four ways:

| Check | Passes when | Fails when |
| --- | --- | --- |
| Manifest match | Your bug maps to a real seeded defect | You report intended behaviour or a non-issue |
| Repro steps | A grader can follow them and reproduce | Steps are vague, skip a step, or do not trigger it |
| Severity | Within one level of the manifest's rating | A crash filed as "minor", or a typo filed as "critical" |
| Duplicates | One report per defect | The same bug filed twice |

A report only counts when all four hold. Ten clean reports beat eighteen sloppy ones.

## Try it — score these three drafts

Imagine the manifest says: *"Checkout: total ignores quantity, severity Major."*

- Draft A: "Total is wrong." — fails repro (no steps, no expected vs actual).
- Draft B: "Add 3 of item X to cart, go to checkout. Expected total = 3 x price; actual = 1 x price. Severity: Minor." — matches manifest, steps reproduce, but severity is understated. One level off Major is within tolerance, so it still counts.
- Draft C: same as B but filed twice with different titles — the second is a duplicate and is dropped.

Draft B is your model: concrete steps, expected vs actual, sane severity.

## Do it — build your route and checklist

On paper, before any clock starts, draft two things for BuggyShop v1.0.

First, a **hunt route** through signup -> login -> browse -> cart -> checkout -> confirmation. For each stop, write one line on what you will probe: boundaries on quantity and price fields, invalid emails and passwords at signup, empty cart at checkout, back-button and refresh mid-flow. Budget time per stop (e.g. 12-15 min) so no single screen eats the hour.

Second, a **reporting checklist** you apply to every bug as you find it: title naming the screen and symptom, numbered repro steps, expected vs actual, severity. Note bugs in a running list and write them up as you go — do not trust memory at minute 80.
