## What it does vs how well it does it

Two big families:

- **Functional testing** — does the feature *do what it should*? Behaviour
  against requirements: login logs you in, the total adds up, the filter filters.
- **Non-functional testing** — *how well* does it do it? Qualities like speed,
  usability, security, and compatibility.

A feature can be perfectly functional and still fail non-functionally: checkout
that works but takes 12 seconds, or a form that validates correctly but is
baffling to use.

## See it — the map (not the depth)

| Non-functional type | Asks |
| --- | --- |
| Performance | Is it fast / does it scale under load? |
| Usability | Is it easy and pleasant to use? |
| Security | Can it be misused or breached? |
| Compatibility | Does it work across browsers/devices/OSes? |
| Reliability | Does it keep working over time / under stress? |

This module gives you the *map*; depth in performance, security, etc. comes in
later tracks. *(A future card explorer will flip each type to a 60-second demo on
BuggyShop — e.g. a usability card showing a confusing checkout.)*

## Try it — classify the complaint

- "The discount never applies." → **functional** (wrong behaviour).
- "The page takes 8 seconds to load." → **non-functional / performance**.
- "I couldn't figure out how to remove an item." → **non-functional / usability**.
- "Other users can see my order history." → **non-functional / security**.

Same product, very different test types — and different specialists.
