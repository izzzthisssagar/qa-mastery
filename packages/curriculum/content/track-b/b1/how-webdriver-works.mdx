## Your code does not click — the browser does

When a Selenium test "clicks a button," your Java code never touches the screen.
It sends an instruction to a **browser driver** — a small standalone program like
**ChromeDriver** or **geckodriver** — and *that* program does the clicking in a
real browser. The driver is the translator between your test and Chrome. Your
code speaks Selenium; the driver speaks Chrome's internal control protocol.

## See it — the request/response model

Your test is a **client**. The driver is a server it talks to. Every step is a
round trip:

Each command is a **request**; the driver performs it in the live browser and
returns a **response**. Your test waits for each response before sending the
next command — which is exactly why test steps run *in order* and why a slow
page can stall the whole test until the response comes back.

## Try it — match the versions

The driver speaks the control protocol of **one specific browser version**. A
ChromeDriver built for Chrome 120 can choke or behave oddly against Chrome 124.
This mismatch is the single most common "my tests broke overnight" cause — the
browser auto-updated, the driver didn't. Modern Selenium ships **Selenium
Manager**, which fetches the matching driver for you, but knowing *why* the pair
must stay in step is what lets you fix it when the automation fails.
