## Thinking in Flows

While techniques like Boundary Value Analysis focus on testing single input fields, **Use Case Testing** focuses on testing the **entire flow** of an application exactly how a real user would interact with it.

A Use Case describes the interaction between an **Actor** (the user or another system) and the **System** to achieve a specific goal. 

## Anatomy of a Use Case

1. **Actor:** Who is doing the action? (e.g. Shopper, Admin, Guest)
2. **Preconditions:** What must be true before this starts? (e.g. User is logged in, item is in stock)
3. **Primary Success Scenario (The Happy Path):** The perfect flow where everything works on the first try.
4. **Alternate Paths / Exceptions:** What happens if the credit card is declined? What if the item goes out of stock mid-checkout?
5. **Postconditions:** What is the state of the system after? (e.g. Order is created, email is sent)

## Creating Tests from Use Cases

To design tests based on a Use Case, you must write at least one test case for:
- The **Happy Path**
- Every identified **Alternate Path**
- Every identified **Exception/Error Path**

This ensures that not only does the core feature work, but the system gracefully handles real-world problems. This technique is highly valued because it directly maps to business requirements and guarantees that the software delivers real value to the user.
